Root cause: cuBLASLt dispatches a 128×128×128 tile for every FP8 linear regardless of M. At M=8, 120/128 rows of the M dimension are padding — SMs are doing useful work on ~6% of the matmul. VRAM use on an 80 GB H100

Memory posture at idle
rvLLM sizes its HBM arena to exactly what the current batch needs (7 GB weights + sized-for-N KV + scratch). vLLM pre-reserves 74 GB for its paged-KV allocator on server start; the actual amount used at any given moment is similar, but the reservation shape is different.

What the freed VRAM buys projected single-card scaling with 72 GB of unused HBM Estimate

Terminology note. Tensor parallelism (TP) is a multi-GPU technique — it splits each layer's weight matrix across devices, with NVLink/NCCL collectives between layers. You can't do TP on one card. On a single H100 with 72 GB free, the useful lever is data parallelism / batch scaling / multi-instance: use that spare HBM to either push batch much higher, load long-context KV, or run several copies of the same model side-by-side. Below are honest projections for each; numbers are extrapolated from the measured N=64–512 sweep unless explicitly measured.

Path 2: multiple Qwen2.5-7B instances on one card (data parallelism)

At 7.8 GB per instance, a single 80 GB H100 fits ~10 independent Qwen2.5-7B engines side-by-side with separate KV pools, independent schedulers, no inter-instance traffic. Each instance sees a fraction of the SMs and HBM bandwidth (approximately 1/k of each when k instances run). At k=4 you'd see roughly 4× request capacity at ~1/4 per-request throughput — same aggregate tok/s, 4× more concurrent sessions. Useful when latency-per-user matters more than total GPU throughput (e.g. many low-QPS chat users).

Path 3: long-context serving on a single H100

At 32 concurrent sequences with 72 GB of free KV:

  • 32 seqs @ 250k context (72 GB / 32 / (28 × 8 × 128 × 1 byte) ≈ 250,000 tokens/seq)
  • 16 seqs @ 500k context
  • 4 seqs @ 2M context (needs RoPE extension; position encoding hold is separate)

vLLM's 74 GB reservation pre-allocates its KV pool at server start; rvLLM's per-batch sizing means you can admit requests with wildly different context lengths without upfront planning.

Path 4: tensor parallelism on a single card — why it fails Estimate

A reasonable question: if we have 72 GB free, can we "simulate multiple GPUs" on one card and do TP? Three ways this gets attempted; each loses for a concrete reason.

1. MIG (Multi-Instance GPU)

H100 supports partitioning into up to 7 MIG instances with dedicated SMs and HBM per slice. You can run TP across MIG instances — hardware-real, not simulated. The problem is inter-MIG communication: it does not use NVLink. It routes through the PCIe root complex, which is ~10–50× slower than NVLink for small collectives. TP needs an all-reduce after each attention layer and after each MLP — 28 layers × 2 = 56 collectives per decode step.

Hypothetical MIG TP-2 step time breakdown at N=128 Estimate
Current rvLLM: 5.83 ms/step (measured). Hypothetical TP-2-on-MIG splits each GEMM in half (compute ~halves to 2.9 ms) but inserts 56 inter-MIG all-reduces. At ~500 μs per small-payload PCIe all-reduce (typical measured on H100 MIG pairs), comm alone is 28 ms. Net step time: ~31 ms (5.3× slower than single-card).

2. Multi-process (CUDA MPS)

Run two processes on one card, each sees the full GPU. Time-multiplexes the SMs. This isn't TP — each process runs the whole model, not a split. It's equivalent to Path 2 (data parallelism / multi-instance). Useful for concurrent capacity, not for one-forward-pass acceleration.

3. Software TP inside one process

Split each weight tensor in half, run two smaller GEMMs sequentially, concatenate. Strictly slower than the combined GEMM: if the original saturated SMs, two halves serialize on the same SMs with launch overhead; if the original was SM-underutilized, the fix is better tile selection (smaller M-tile), which is what our v0.3.1 top-16 cuBLASLt autotune does — not TP.

Why TP even exists the real math

TP pays off on multi-GPU because two benefits compound:

  • Memory split — weights that don't fit on one card (70B+ FP8 = 70 GB, won't fit in 80 GB after KV + activations) are shared across cards.
  • Independent compute — each card has its own full SM pool and HBM. Work runs genuinely in parallel, limited only by the collective latency.

On one card: neither applies. 7B fits. One SM pool. The collective cost is the only term that scales, and it scales up.

What the numbers say. Small ops cluster at 125–200 μs — that's the XLA host-to-device-dispatch floor on a single v6e chip, not the kernel body. Anything bigger (paged_attention at 1.8 ms, the full decode layer at 2.15 ms) is real compute. The paged-attention cost is dominated by the stablehlo.gather on the block table + 8× GQA expansion; this is the same math the H100 FA3 kernel is doing, but implemented in XLA primitives rather than a hand-tuned CUDA kernel. The 6 FP8 rows are the honest gap: TPU v6e has no StableHLO FP8 op yet and we refuse to insert a silent bf16 fallback. Fixing them is a StableHLO spec problem, not a port problem.
TPU suite (new, under tpu/): JAX traces each CUDA kernel's logical signature via a ShapeDtypeStruct trace-spec, jax.jit(...).lower(...).compiler_ir(dialect="stablehlo") emits one .mlir per kernel (1,138 lines total), and harness/execute_all.py runs them on the active JAX backend with warmup×3 + median-of-20 iter timing. On the host (M5) this proves lowering; the above numbers are on a fresh Cloud TPU v6e-1 in us-east5-b, deployed via harness/deploy_to_tpu.sh, then the VM was torn down.

Method — how these runs were produced

rvLLM 0.3.0 (tag v0.3.0, commit 82b1f59):
RVLLM_MODEL_DIR=... RVLLM_KERNELS_DIR=... RVLLM_CUTLASS_SO=... RVLLM_FA3_SO=... RVLLM_POLICY=... RVLLM_TTFT=1 RVLLM_REAL_PREFILL=1 RVLLM_PREFILL_LEN=16 RVLLM_BATCH=<N> RVLLM_ITERS=128 RVLLM_WARMUP=5 ./target/release/rvllm-bench

Cold TTFT = first timed prefill call after 5 warmup decode steps. Hot TTFT = second timed prefill. Throughput = (ITERS × N) / (timed decode window) after warmup.

vLLM 0.19 V1:
vllm serve Qwen2.5-7B-Instruct --quantization fp8 --kv-cache-dtype fp8_e4m3 --dtype bfloat16 --max-model-len 2048 --max-num-seqs 512 --max-num-batched-tokens 16384 --gpu-memory-utilization 0.9

Then, for each N, run twice (cold then hot) on the same server process:
vllm bench serve --dataset-name random --num-prompts <N> --max-concurrency <N> --random-input-len 16 --random-output-len 512 --ignore-eos

Reported throughput = tool's Output token throughput (tok/s). Reported TTFT = tool's Mean TTFT.

nsys profile (N=8, rvLLM): nsys profile --trace=cuda,nvtx,osrt --sample=none --cudabacktrace=none --output=rvllm_n8 --stats=true rvllm-bench. GPU was idle (no vLLM). Kernel sums parsed over the 21 steady-state graph-replay steps post-warmup.
