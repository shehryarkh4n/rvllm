How 1:1 is the vLLM comparison?

An honest breakdown of what's the same, what's different, and why the 16% is real.

What's genuinely the same

  • Same H100 SXM 80GB, same Qwen2.5-7B weights from HuggingFace, same FP8 E4M3 quantization type, same batch size (N=128), same CUDA graphs enabled.
  • Both engines report steady-state decode tokens per second. vLLM's number is their engine log line Avg generation throughput, which is measured only during the decode phase — prefill is excluded from this metric.
  • Both use Hopper-optimized attention (vLLM via FlashInfer, v3 via FlashAttention-3 built from source).
  • Both capture the forward loop into a CUDA graph and replay it.

The three original caveats — and how v3's bench now addresses each

  1. vLLM does real prefill (16 input tokens × 128 prompts = 2,048 tokens of real prompt processing before decode starts). Fix: v3 now does real prefill too under RVLLM_REAL_PREFILL=1 — one multi-query causal FP8 attention call via FA3 paged-prefill over the concatenated prompt (total_q = N × prompt_len) with varlen cu_seqlens_q. Both engines' Avg generation throughput is measured during the decode phase, under identical KV-populated state.
  2. vLLM runs its scheduler every step. Even at steady-state batch=128, it checks for request state, completions, block table management. Our graph was frozen. Fix: v3's bench now re-uploads positions, slot_mapping, and context_lens every iteration inside the timed loop, so v3 pays the same per-step HtoD cost that vLLM's scheduler pays for metadata management.
  3. vLLM's KV cache has real content from real prompts. Ours had post-warmup garbage. Fix: v3's bench now runs the real FA3 paged-prefill before the timed window, so paged KV is populated with legitimate rope+quant activations — matching what vLLM measures under.

Net impact of all three fairness fixes: −0.3% at N=128; final v3 throughput with real prefill is 22,069 / 34,364 / 42,030 tok/s at N=128/256/512. The per-step metadata upload is cheap against the 28-layer forward + LM head compute, so the +13.8% edge at N=128 — and the +16.4% / +22.8% edges at N=256/512 — are genuine improvements, not measurement tricks.

Also different but unchanged by these fixes: vLLM uses FlashInfer for attention; v3 uses FlashAttention-3 built from source. vLLM's FP8 path is its own; v3 quantizes weights at load time to FP8 E4M3 with a clamp-ppm gate and routes every linear through cublasLtMatmul with fused bias / residual epilogues.

What this comparison tells you

At the decode kernel level, v3 is faster. 14–23% faster steady-state tokens per second on the same GPU, same model, same quantization, both with CUDA graphs. That delta comes from four structural choices (fused QKV, cuBLASLt for every linear, fused epilogues, no Python dispatch).

In a full serving scenario, the gap would narrow. v3 now runs real prefill but continuous batching and a full scheduler are still future work. Some of the 14–23% advantage would pay for that scheduler overhead. The comparison is legitimate for kernel-throughput and prompt-ingest, not yet for end-to-end multi-request serving.

Kernels

Every kernel has a pinned variant and a workspace contract. No dispatch fallback chains.

cuBLASLt FP8 (v3 fast path)

v3/crates/rvllm-cutlass/src/cublaslt.rs

All 5 FP8 linears (QKV, O, gate_up, down, lm_head) route through cublasLtMatmul. One library call handles the matmul plus the epilogue:

  • QKV: CUBLASLT_EPILOGUE_BIAS fuses the f16 row-broadcast bias into the GEMM. Replaces (GEMM + add_bias_kernel) with one launch.
  • O / down: α=1, β=1 with the residual tensor as C; the residual-add lands inside the GEMM epilogue.
  • gate_up / lm_head: plain FP8 GEMM, per-tensor A/B scales, f16 output.
  • Per-tensor scales (A_SCALE_POINTER, B_SCALE_POINTER) are device pointers; cuBLASLt's per-shape heuristic picks the algo with no custom policy.json entry.

CUTLASS SM90 FP8 GEMM (archive path)

kernels/cutlass_fp8_gemm.cu, kernels/cutlass_fp8_gemm_residual.cu

40 non-residual variants + 10 residual-fused variants, templated on (TileShape, ClusterShape, KernelSchedule). v2 ran here, v3 still loads the .so for the variant sweep harness but does not dispatch through it for decode.

  • Schedule rule: mainloop schedule must match epilogue schedule. Mismatched variants are typed CutlassError::SchedulePairingInvalid at load time.

FlashAttention-3 SM90 paged decode

kernels/fa3_sm90_wrapper.cu → libfa3_kernels.so (23 MB)

WGMMA + TMA, built from FlashAttention-3 Hopper source. Paged KV layout [2, num_blocks, block_size, num_kv_heads, head_dim], GQA via num_heads / num_kv_heads ratio. head_dim = 128 hard gate.

No PTX fallback. If libfa3_kernels.so is not present, the engine refuses to load.

    fused_add_rmsnorm_fp8_quant                mlp norm + FP8 quant
    cublasLtMatmul(FP8)                         gate_up proj
    fused_silu_mul_fp8_quant                   SiLU(gate)*up + FP8 quant
    cublasLtMatmul(FP8, beta=1 residual)        down proj + residual add

12 x 28 + 3 = 339 launches, all captured into one cuGraphLaunch. Each call fuses one recognizable composite. No megakernels. The only GEMM-fused epilogues are the ones cuBLASLt gives for free (bias / residual-add).

Correctness discipline

Explicit rules. Violations fail the build or startup — never degrade silently.

  1. No fallbacks. Missing autotune policy entry for a shape panics with the shape. Missing FA3 .so refuses startup. Missing CUTLASS .so refuses startup. Stale autotune caches from prior deploys are not consulted — policy lives in the SHA-pinned build artifact.
  2. Graph-capture invariant. Metadata buffer layout is frozen per (bucket, max_blocks_per_seq). Captured graphs bind those exact offsets. There is no "non-padded" upload path. Prefill and decode have separate APIs; they do not share metadata offsets. This is the fix for the April 16 CUDA_ERROR_ILLEGAL_ADDRESS regression.
  3. Real block-change detection. Scheduler emits ContinuedRequest::block_table_update: Option<Vec<BlockId>> whenever a sequence's physical block list grows. Worker unions this with CoW copies. Missing either signal leaves stale block_ids in the captured graph — wrong KV reads, silent correctness bug, no crash.
  4. CUTLASS schedule/epilogue pairing. Mainloop and epilogue must match. Mismatched variants cause CUDA_ERROR_ILLEGAL_ADDRESS only inside graph replay. Enforced in v3 as a CUDA static_assert. In v2, dispatch pins the fused residual path to variant v1 (Coop/Coop, hand-verified).
  5. No unwrap() in libraries. Result<T, RvllmError> end-to-end. Errors carry structured context (stream, kernel name, launch config) — not stringified DriverError that loses fault location.

The cuBLASLt move was the dominant per-token win. FP8 KV is the capacity unlock: the 2× KV-memory reduction enabled N=256 and N=512, where GEMMs saturate HBM and absolute throughput doubles. The Phase F scratch-buffer fixes also lifted steady-state throughput at every batch (+6% at N=128, +10% at N=256, +4% at N=512) because the prior sizing silently truncated prefill activations.

Reproduce (v3)

One H100 box, everything from source

# One-time kernel artifacts (~15 min)
bash kernels/build.sh               # fused PTX (rmsnorm, rope, silu, argmax, ...)
bash kernels/build_cutlass_so.sh    # libcutlass_kernels.so (FP8 variants)
bash kernels/build_fa3.sh           # libfa3_kernels.so

# v3 manifest + policy (sha-pinned)
python3 v3/kernels/make_manifest.py /workspace/rvllm/kernels/sm_90 sm_90 $(git rev-parse HEAD)
python3 v3/kernels/make_policy.py   /workspace/rvllm/kernels/sm_90/policy.json $(git rev-parse HEAD)

# Build v3 bench
cargo build --release --features cuda --manifest-path v3/Cargo.toml -p rvllm-bench

# Run
RVLLM_MODEL_DIR=/workspace/models/qwen25-7b-instruct \
RVLLM_KERNELS_DIR=/workspace/rvllm/kernels/sm_90 \
RVLLM_CUTLASS_SO=/workspace/rvllm/kernels/sm_90/libcutlass_kernels.so \
RVLLM_FA3_SO=/workspace/rvllm/kernels/sm_90/libfa3_kernels.so \
RVLLM_POLICY=/workspace/rvllm/kernels/sm_90/policy.json \
RVLLM_BATCH=128 RVLLM_ITERS=30 RVLLM_WARMUP=5 \
  ./v3/target/release/rvllm-bench
