Abstract

Andy Norris (m0at) -- San Francisco, April 14, 2026

rvLLM is a from-scratch Rust implementation of the vLLM inference engine for OpenAI-compatible API serving with CUDA GPU acceleration. 310 commits, 31 crates, ~76K lines of Rust, 253 source files. The current tree has 54 CUDA kernels, a Rust-native PTX compiler that generates fused kernels 2-7.5x faster than the hand-written CUDA versions in the measured decode microbenchmarks, FA3 v3 cp.async + split-KV attention, no-fallback kernel validation, and 13 model architectures including Llama, Qwen2, Mistral, DeepSeek-V2, Mixtral, NemotronH MoE, and Gemma.

The v2 engine with CUTLASS SM90 FP8 GEMMs reaches 2.2x vLLM 0.19.0 FP8 on Qwen2.5-7B at N=64 (29,868 tok/s vs 13,663 tok/s) and 1.3x at N=128 (30,076 tok/s vs 22,618 tok/s) on H100 SXM 80GB. At batch-1 with 128 output tokens, rvLLM delivers 13,865 tok/s vs vLLM's 175.3 tok/s (79x), reflecting the difference between a compiled Rust engine and Python-framework overhead on small batches. FP8 and F16 converge at ~30K tok/s because decode is memory-bandwidth-bound on H100 HBM3 (3.35 TB/s).

The FP8 inference stack uses CUTLASS 3x SM90 FP8 GEMMs (Hopper WGMMA + TMA, same kernel family as vLLM), fused RMSNorm + per-token FP8 quantize kernels, and fused SiLU*mul + per-token FP8 quantize kernels. Each fused kernel replaces 3 separate kernels, bringing the total to 8 kernels per transformer layer (down from 15 with the cuBLASLt path). A scheduler preemption bug that caused N=96+ throughput cliffs has been fixed.

The API layer implements 14 OpenAI-compatible endpoints including completions, chat completions with tool/function calling, the Responses API (stored multi-turn conversations, background execution, streaming argument deltas), embeddings, batch inference, and a Prometheus metrics endpoint. Weight loading supports SafeTensors (single and sharded) and GGUF (Q4_0, Q4_K_M, Q5_0, Q5_K_M, Q8_0). Guided decoding enforces JSON, JSON Schema, and regex constraints at the token level. Sampling uses a Zig SIMD backend for hot paths (softmax, argmax, weight conversion) with 1.4-6.3x speedups over scalar Rust.

JIT Kernel Compiler

Rust-native PTX generation -- no nvcc, no Triton, no Python

The rvLLM fusion compiler (crates/rvllm-fusion/src/ptx_emit.rs) emits PTX assembly directly from Rust at model load time. Kernels are shape-specialized with vectorized loads, warp shuffle reductions, and shared memory tiling tuned for the exact model dimensions. No external compiler dependency.

What Changed

CUTLASS FP8, fused kernels, scheduler fix, vectorized SiLU+FP8, stream-K/split-K autotune

  1. CUTLASS SM90 FP8 GEMMs replaced cublasLt. Same kernel family vLLM uses (Hopper WGMMA + TMA), but driven directly from Rust via the CUTLASS 3x C++ API. Eliminates the cublasLt wrapper overhead and autotuning warmup.
  2. Fused RMSNorm + per-token FP8 quantize. A single kernel reads the hidden state, normalizes, quantizes to FP8 E4M3 per-token, and writes the result. Replaces 3 separate kernels (norm, abs-max, quantize).
  3. Fused SiLU*mul + per-token FP8 quantize. Same pattern for the MLP activation: gate activation, element-wise mul, quantize in one kernel. Replaces 3 separate kernels.
  4. 8 kernels per transformer layer. Down from 15 with the old cublasLt path. Fewer kernel launches, fewer global memory round-trips.
  5. Scheduler preemption fix. The watermark-triggered eviction was firing with no admission pressure, causing a throughput cliff at N=96+. Fixing it let throughput scale smoothly to N=128.
  6. Vectorized fused SiLU*mul + FP8 quantize (April 14). 128-bit loads (uint4) and 64-bit FP8 stores (uint2) with register caching. Eliminates second-pass global memory reads in the MLP activation kernel.
  7. Stream-K / split-K FP8 GEMM autotune (April 14). 7 new CUTLASS variants (v25-v31) with StreamKScheduler and explicit K-decomposition. Split-K=4 on Down projection: 28.5% speedup by going from 21% to 85% SM utilization at M=64.
  8. FP8 autotune expansion (April 14). 32 FP8 GEMM variants total with standalone autotune binary. FP8FastAccum schedule aliases. +12-15% decode throughput across batch sizes.
The result: rvLLM v2 FP8 and F16 both converge at ~30K tok/s, proving the engine is now memory-bandwidth-bound on H100 HBM3 (3.35 TB/s). The GEMM dtype no longer matters for decode throughput at these batch sizes.

Three-Tier Kernel Stack

JIT > Hand-written CUDA > cuBLAS

  1. JIT fused kernels (fastest) -- Rust PTX emitter generates shape-specialized fused kernels at model load. 2-7.5x faster than hand-written CUDA for M=1 decode. Patterns: RMSNorm+GEMV, Add+RMSNorm+GEMV, SiLU*Mul+GEMV.
  2. 54 hand-written CUDA kernels -- fused decode (add+norm+QKV+bias, RoPE+cache, GQA attention, O-proj+gateup, silu+down), FP8 E4M3 variants, TMA async-prefetch GEMV, WGMMA tensor core GEMV, split-KV attention, INT4 W4A16 GEMV (4 fused variants, pending Rust wiring). No-fallback validation at startup ensures every kernel loads or the server fails fast.
  3. cuBLAS/cublasLt (batched QKV/O/down) -- autotuned algorithm selection with vendored type shim for cudarc 0.19. 32 candidates benchmarked per GEMM shape at startup.
  4. CUTLASS (batched GateUp+SiLU) -- the current default hybrid epilogue path for the MLP expansion.

Plus an LLVM NVPTX backend (experimental) -- full compiler pipeline: Fusion IR -> LLVM IR -> NVPTX -> PTX via inkwell. Same backend as Triton.

Where the Speed Comes From

Compound effect of fifteen reinforcing mechanisms

  1. JIT fused kernels -- compiler-generated fused norm+GEMV eliminates HBM round-trips
  2. GPU-resident argmax -- eliminates 74 MB DtoH per step
  3. CUDA graph replay -- 35 pre-captured graphs, 1.7 sec total capture time
  4. Batch-1 dispatch fix -- normal single-stream decode now takes the right default path
  5. Explicit batched hybrid GEMM routing -- QKV/O/down on cuBLAS, GateUp+SiLU on CUTLASS
  6. cuBLAS autotuning -- best algorithm per GEMM shape, benchmarked at startup
  7. FP16 hgemm + f16 KV cache -- tensor cores for all projections
  8. FA3 v3 cp.async + split-KV attention -- cp.async bulk copies, split-KV across thread blocks, parallel softmax, bank-conflict-free smem, V reuse across head groups
  9. No GIL, no GC -- deterministic Rust, parallel scheduling via Rayon
  10. CUTLASS SM90 FP8 GEMMs -- Hopper WGMMA + TMA, same kernel family as vLLM, driven from Rust via CUTLASS 3x C++ API
  11. Fused RMSNorm + FP8 quantize -- single kernel replaces norm + abs-max + quantize (3 kernels down to 1)
  12. Fused SiLU*mul + FP8 quantize -- single kernel replaces activation + mul + quantize (3 kernels down to 1)
  13. 8 kernels per layer -- down from 15 with cublasLt path, fewer launches, fewer HBM round-trips
  14. Zig SIMD sampling -- NEON/AVX-512 softmax, argmax, weight conversion (1.4-6.3x)
  15. Chunked prefill scheduler -- bounds prefill tokens per step, prevents decode starvation
  16. Scheduler preemption fix -- eliminates watermark-triggered eviction cliff at N=96+
  17. Vectorized fused SiLU+FP8 -- uint4 128-bit loads, register caching, uint2 64-bit FP8 stores (8 FP8 values per store)
  18. Stream-K / split-K autotune -- CUTLASS StreamKScheduler decomposes K across SMs; split-K=4 gives 85% SM utilization on memory-bound shapes
  19. FP8 GEMM autotune -- 32 variants benchmarked per shape, standalone binary populates cache
