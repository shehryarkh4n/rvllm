rvLLM: A High-Performance LLM Inference Engine in Rust

m0at

2026

Abstract

We present rvLLM, a high-performance large language model inference engine written in Rust. The system is structured as a 23-crate workspace covering the full inference stack -- from tokenization and model loading through GPU-accelerated attention, KV-cache management, speculative decoding, and an OpenAI-compatible API server. rvLLM integrates 10 custom CUDA kernels and a cuBLAS wrapper for matrix operations, while leveraging Rust's ownership model and rayon for safe CPU parallelism. Preliminary benchmarks show significant throughput improvements over Python-based inference engines, with lower tail latencies and reduced memory overhead. We describe the architecture, key design decisions, CUDA integration strategy, and early performance results.

1. Introduction

Large language model inference at scale presents a systems engineering challenge that spans memory management, GPU kernel optimization, scheduling, and network I/O. The dominant open-source inference engines -- vLLM, TGI, and others -- are implemented primarily in Python, relying on PyTorch and custom CUDA extensions for performance-critical paths. While effective, this approach carries inherent costs: Python's GIL constrains CPU-side throughput, dynamic dispatch and garbage collection introduce latency variance, and the boundary between Python orchestration and C++/CUDA computation creates optimization barriers.

rvLLM takes a different approach: the entire inference stack is implemented in Rust, from the API server down to the GPU memory allocator. Rust's ownership and type system provide compile-time guarantees around memory safety and data-race freedom without runtime overhead. The rayon crate enables work-stealing parallelism for CPU-bound operations, while direct CUDA FFI bindings eliminate the Python-to-C++ bridge entirely.

This paper describes the design, implementation, and preliminary evaluation of rvLLM. We focus on the architectural decisions that differentiate it from existing engines, the CUDA integration strategy, and early benchmark results comparing throughput, latency, and memory efficiency against Python vLLM.

2. Architecture

rvLLM is organized as a Cargo workspace containing 23 crates. Each crate encapsulates a well-defined responsibility, enabling independent compilation, testing, and -- where applicable -- replacement. The workspace structure reflects the logical layers of an inference engine:

2.1 Core Infrastructure

  • rvllm-core -- shared types, error handling, and trait definitions
  • rvllm-config -- configuration parsing and validation
  • rvllm-telemetry -- structured logging and metrics collection

2.2 GPU and Memory

  • rvllm-gpu -- CUDA device management, kernel launching, cuBLAS bindings
  • rvllm-memory -- GPU and CPU memory allocators with pool-based allocation
  • rvllm-block-manager -- paged memory management for KV-cache blocks
  • rvllm-kv-cache -- key-value cache implementation with block-level granularity
  • rvllm-attention -- attention kernels (PagedAttention, FlashAttention-2 dispatch)

2.3 Model Pipeline

  • rvllm-tokenizer -- tokenizer bindings (HuggingFace tokenizers via FFI)
  • rvllm-model-loader -- safetensors/GGUF loading, weight sharding
  • rvllm-model-runner -- forward pass orchestration
  • rvllm-quant -- quantization support (GPTQ, AWQ, INT8, FP8)
  • rvllm-sampling -- top-k, top-p, temperature, repetition penalty

2.4 Scheduling and Execution

  • rvllm-sequence -- sequence state management and lifecycle
  • rvllm-scheduler -- continuous batching with preemption support
  • rvllm-worker -- per-GPU worker threads
  • rvllm-executor -- multi-GPU execution coordination
  • rvllm-speculative -- speculative decoding with draft model support

2.5 Server Layer

  • rvllm-engine -- top-level engine tying scheduler, executor, and model runner
  • rvllm-api -- OpenAI-compatible REST API definitions
  • rvllm-server -- HTTP server (Axum-based) with streaming SSE support
  • rvllm-python -- PyO3 bindings for Python interop
  • rvllm-bench -- benchmarking harness and load generation

The crate dependency graph is strictly layered: server crates depend on engine crates, which depend on execution crates, which depend on GPU and model crates, which depend on core infrastructure. Circular dependencies are prevented by Cargo's module system. This layering enables, for instance, running the engine without the HTTP server for embedded use cases, or swapping the attention implementation without touching the scheduler.

3. CUDA Integration

The rvllm-gpu crate provides Rust bindings to CUDA and cuBLAS through a combination of bindgen-generated FFI and hand-written safe wrappers. All GPU memory is managed through Rust's ownership system: device allocations are represented as types that implement Drop, ensuring deallocation on scope exit even in the presence of early returns or panics.

3.1 Custom Kernels

rvLLM includes 10 custom CUDA kernels, compiled with nvcc and linked via build.rs:

  1. PagedAttention v2 -- block-sparse attention with variable sequence lengths
  2. FlashAttention-2 dispatch -- tiled fused softmax-matmul kernel
  3. RoPE embedding -- rotary positional encoding applied in-kernel
  4. RMSNorm -- fused root-mean-square normalization
  5. SiLU activation -- fused SiLU (Swish) with gate projection
  6. Quantized GEMM -- INT4/INT8 matrix multiply with dequantization
  7. KV-cache copy -- block-to-block cache migration for preemption
  8. Token embedding -- batched embedding lookup with padding masking
  9. Sampling kernel -- fused top-k/top-p with temperature on GPU
  10. Layernorm -- fused layer normalization with residual add

3.2 cuBLAS Integration

Dense matrix multiplications are dispatched through cuBLAS via a Rust wrapper that manages cuBLAS handles per-stream and selects the optimal GEMM algorithm at initialization via cublasGemmAlgo_t auto-tuning. The wrapper exposes a safe API: 

pub fn gemm(
    &self,
    a: &GpuTensor,
    b: &GpuTensor,
    c: &mut GpuTensor,
    alpha: f32,
    beta: f32,
) -> Result<()> {
    // Validates dimensions at runtime, dispatches to cublasGemmEx
    // with auto-selected algorithm and compute type
    unsafe { self.raw_gemm(a, b, c, alpha, beta) }
}

3.3 Memory Management

GPU memory is managed through a two-level allocator. A slab allocator handles large, long-lived allocations (model weights, KV-cache blocks), while a bump allocator serves short-lived activation tensors within a single forward pass. The bump allocator resets between iterations, eliminating fragmentation for transient allocations. Both allocators are implemented in Rust, calling cudaMalloc/cudaFree only at the slab level.

4. Key Design Decisions

4.1 Zero-Copy Where Possible

rvLLM minimizes data copies between the CPU and GPU. Tokenized input is written directly into a pinned host buffer and transferred to the GPU via a single cudaMemcpyAsync. Output logits are sampled on-device; only the final token IDs are copied back. This reduces PCIe traffic to the minimum necessary.

4.2 Continuous Batching with Preemption

The scheduler implements continuous batching: new requests are admitted into a running batch without waiting for the longest sequence to complete. When GPU memory pressure exceeds a configurable threshold, the scheduler preempts low-priority sequences by swapping their KV-cache blocks to CPU memory, resuming them when capacity is available. Block-level preemption avoids recomputation.

4.3 Async I/O Throughout

The server layer uses tokio for async HTTP handling and SSE streaming. The engine communicates with workers via lock-free channels (crossbeam). CUDA operations are issued on dedicated streams and synchronized via events, allowing the CPU to prepare the next batch while the GPU executes the current one.

4.4 Speculative Decoding

The rvllm-speculative crate implements speculative decoding with a configurable draft model. The draft model generates k candidate tokens, which the target model verifies in a single forward pass. Accepted tokens skip individual decode steps, improving throughput for sequences where the draft model has high acceptance rates. The implementation handles the token tree and rejection sampling entirely on the GPU.

6. Related Work

vLLM [1] introduced PagedAttention and continuous batching for LLM serving, implemented in Python with custom CUDA extensions. rvLLM adopts the same PagedAttention algorithm but implements the entire stack in Rust, eliminating the Python runtime overhead.

TensorRT-LLM [2] provides high-performance inference through TensorRT's graph optimization and kernel fusion. It achieves excellent throughput but requires model conversion to TensorRT engines, limiting flexibility. rvLLM loads standard safetensors and GGUF formats directly.

llama.cpp [3] demonstrates that C/C++ inference engines can achieve strong performance on consumer hardware through aggressive quantization. rvLLM targets the server-side inference use case with continuous batching and multi-GPU support, complementing llama.cpp's single-user focus.

Text Generation Inference (TGI) [4] by Hugging Face uses Rust for its HTTP server layer but delegates model execution to Python/PyTorch. rvLLM extends the Rust boundary to encompass the full inference pipeline.

candle [5] provides a Rust-native tensor library with CUDA support. While candle focuses on the tensor computation layer, rvLLM implements the full serving stack including scheduling, batching, and KV-cache management on top of direct CUDA kernel calls.

7. Conclusion

rvLLM demonstrates that a full LLM inference engine can be effectively implemented in Rust, from CUDA kernel dispatch to HTTP streaming. The 23-crate workspace provides clear separation of concerns while Rust's type system enforces memory safety across GPU and CPU boundaries without runtime cost.

Preliminary results suggest meaningful improvements in throughput, tail latency, and memory efficiency compared to Python-based engines. The elimination of the Python runtime removes an entire class of performance variability, while Rust's rayon-based parallelism enables CPU-side scaling that the GIL prevents in Python.

Future work includes expanding multi-GPU support via NCCL integration, implementing tensor parallelism across nodes, adding support for mixture-of-experts architectures, and completing the benchmark suite with production-scale workloads.
